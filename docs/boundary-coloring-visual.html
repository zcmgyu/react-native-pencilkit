The "Always-Empty Canvas" Architecture

The key insight: PencilKit kills the active stroke if you set canvasView.drawing during a touch. So we never swap drawings. Instead, the canvas is always empty when a new stroke starts.

1. Touch Down

ZoneTouchDetector fires, detects which zone was touched, applies the zone's mask to the canvas. Then immediately fails (state = .failed) so PencilKit gets the touch.

2. PencilKit Starts Stroke

PencilKit processes the same touch and begins drawing. The stroke is visually clipped by the CALayer.mask on the canvas — it only appears inside the zone.

Note: During drawing, the stroke is a temporary overlay. It's NOT in canvasView.drawing.strokes yet.

3. Finger Lifts

canvasViewDidEndUsingTool fires. But the stroke is STILL not in drawing.strokes!

4. Stroke Finalized

canvasViewDrawingDidChange fires with increased stroke count. This is the commit moment.

  • Render stroke via PKDrawing.image() (reliable, view-independent)
  • Mask it to the zone using Core Graphics destinationIn blend
  • Composite onto coloredLayer
  • Clear the canvas (canvasView.drawing = PKDrawing())

5. Ready for Next Stroke

Canvas is empty again. Next touch in ANY zone starts instantly — no swap needed.

PencilKit Delegate Timing (Critical Discovery)

PencilKit's delegate methods fire in a non-obvious order. Getting this wrong caused 10+ failed implementations.

Touch Down:
  1. PKCanvasView internal recognizers fire → didBeginUsingTool
  2. ZoneTouchDetector.touchesBegan fires → zone detection → mask applied

During Drawing:
  3. canvasViewDrawingDidChange fires (stroke path growing)
     BUT: stroke is NOT in drawing.strokes yet! It's a temporary overlay.

Finger Lifts:
  4. canvasViewDidEndUsingTool fires
     BUT: stroke is STILL not in drawing.strokes!

  5. canvasViewDrawingDidChange fires (stroke count increases)
     NOW: stroke IS in drawing.strokes. This is the ONLY safe commit point.

Step 1: Convert to Grayscale

The coloring page can be any format (JPEG, PNG, etc.) with any color depth. We convert it to a simple 8-bit grayscale image — one byte per pixel, values 0 (black) to 255 (white).

Original image
→
8-bit grayscale

In code: CGContext with CGColorSpaceCreateDeviceGray() renders any image format into a flat byte array.

Step 2: Threshold (Binary Image)

Each pixel is classified as either white (colorable) or black (outline) based on a threshold (default: 128).

Grayscale
→
Binary: white vs black
pixel > 128 → white (colorable)
pixel ≤ 128 → black (outline / boundary)

The threshold is configurable via the boundaryThreshold prop. Higher values mean thicker outlines (more pixels treated as black).

Step 2.5: Outline Dilation (Gap Bridging)

Problem: If a coloring page has small gaps in its outlines (the line didn't fully close), CCL treats both sides of the gap as one connected region. The user could color across the gap into the neighbor.

Without dilation: leak
→
With dilation: sealed

Solution: Before CCL, expand every black (outline) pixel by outlineDilation pixels (default: 3) in all directions. This bridges gaps smaller than 6 pixels (2 × 3).

Algorithm: Chebyshev distance transform (O(n)):
1. Compute distance from each white pixel to nearest black pixel
2. If distance ≤ outlineDilation → pixel becomes black (outline)

This thickens all outlines uniformly. The colored area shrinks slightly (by outlineDilation pixels from each edge), but the gap is sealed. Increase the value for images with larger gaps.

Step 3: Connected Component Labeling (CCL)

This is the key algorithm. It scans every white pixel and groups connected ones into zones. Each zone gets a unique ID.

Interactive: Watch CCL Scan

Click "Run CCL" to watch the algorithm scan a small grid. White cells are colorable. The scanner assigns zone IDs (colors) to connected groups.

How it works:
1. Scan left→right, top→bottom
2. For each white pixel, check up and left neighbors
3. No labeled neighbor → assign new zone ID
4. One labeled neighbor → copy that zone ID
5. Both neighbors, different IDs → copy smaller, union them
6. Pass 2: flatten all labels to their root
4-connectivity vs 8-connectivity:
4-conn (used)
8-conn (not used)
4-connectivity checks only up/down/left/right.
Diagonal-only touching = separate zones.

Step 4: The Region Map

After CCL, every pixel in the image has a zone ID:

  • 0 = outline pixel (black line — not colorable)
  • 1, 2, 3, ... = zone IDs (each connected white area gets a unique number)

→

Stored as a flat Int32 array: regionMap[y * width + x] = zoneID. For a 1030×1207 image, this is ~5MB.

Step 5: Canvas-Resolution Mapping

The boundary image (e.g., 1030×1207 pixels) is displayed scaled to fit the canvas (e.g., 360×360 points × 3x = 1080×1080 pixels). We pre-compute a mapping from each screen pixel to its zone ID.

For each canvas pixel, we compute: imageX = (canvasX - offsetX) / renderW * imageW. The result is stored as per-zone pixel index lists — to generate a mask for zone 5, just set those specific pixels to opaque. No scanning needed.

The conversion accounts for aspect-fit scaling — the image may be letterboxed/pillarboxed within the canvas. Touching the letterbox area returns zone 0 (no zone).

Mask Generation & Performance

Pre-computed Pixel Indices

After building the canvas map, each zone has a list of pixel indices that belong to it. To generate a mask for zone 5:

  1. Allocate a transparent RGBA buffer (all zeros = invisible)
  2. For each index in zone 5's list: set that pixel to RGBA(255, 255, 255, 255) (opaque white)
  3. Create a CGImage from the buffer

This only touches the zone's own pixels (typically 1-5% of the canvas). Result: ~1ms per mask vs ~50ms for a full scan.

The mask is used as canvasView.layer.mask. CALayer.mask uses the alpha channel: alpha=255 = visible, alpha=0 = hidden.

destinationIn Blend Mode

Core Graphics blend mode that keeps destination pixels only where the source has alpha > 0.

  • Draw the stroke image (destination)
  • Draw the zone mask on top with .destinationIn
  • Result: stroke pixels are kept only inside the zone. Everything outside becomes transparent.

This is more reliable than CALayer.mask for offline rendering because it doesn't depend on the view's display state.

Undo / Redo

PencilKit's native UndoManager resets when canvasView.drawing is set (which happens during commit). So we implement custom undo:

  • Before each commit: Push the current coloredLayer.image onto coloredSnapshots (undo stack)
  • Undo: Pop from undo stack, push current to redo stack, restore coloredLayer.image
  • Redo: Pop from redo stack, push current to undo stack, restore coloredLayer.image
  • New stroke: Clears the redo stack (drawing after undo invalidates redo history)

ZoneTouchDetector

A custom UIGestureRecognizer that captures the touch point and immediately fails.

override func touchesBegan(_ touches, with event) {
    let point = touches.first?.location(in: view)
    onTouchDown?(point)  // Detect zone, apply mask — synchronous
    state = .failed      // Release touch immediately
}

Why .failed? A gesture recognizer in .began or .changed state "holds" the touch, interfering with PencilKit's internal recognizers. By failing immediately, PencilKit processes the touch as if no other recognizer exists.

Previous approach (UILongPressGestureRecognizer) caused canvasViewDidEndUsingTool to fire at the wrong time — on the NEXT touch instead of on finger lift.

Failed Approaches (10+ Iterations)

#ApproachWhy It Failed
1 Accumulated CALayer.mask Strokes overflow into neighbor zones when mask reveals multiple zones
2 Bake in didEndUsingTool Stroke NOT in drawing.strokes yet at that point — renders empty image
3 UILongPressGestureRecognizer Interferes with PencilKit delegates — didEndUsingTool fires on NEXT touch
4 Per-zone drawing swap (sync) Setting canvasView.drawing kills the active stroke — requires two touches
5 Per-zone drawing swap (async) Deferred swap still kills in-progress stroke — just delayed by one frame
6 Swap in touchesBegan PencilKit's didBeginUsingTool fires BEFORE our recognizer — swap too late
7 PKStroke.mask (UIBezierPath) Modifying strokes requires setting canvasView.drawing — same swap problem
8 Large erosion (20px) Visible gap between color and outline. Wide brushes still overflow past the gap.
9 drawHierarchy / CALayer.render Unreliable with CALayer.mask — sometimes captures empty/wrong content
10 PKDrawing.image + wrong timing Correct rendering technique, but called in didEndUsingTool (stroke not ready)
Solution

PKDrawing.image() + destinationIn blend + commit in canvasViewDrawingDidChange

The winning combination: reliable rendering (PKDrawing.image), reliable masking (destinationIn), and the correct timing (drawingDidChange when stroke count increases).
